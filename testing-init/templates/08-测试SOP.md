# 测试 SOP（testing-run 固定执行流程）

本文件由 `testing-init` 生成。`testing-run` 每次复杂测试都必须严格按此顺序执行，不跳步、不临时改命令。简单改动（单元测试能直接覆盖的）不需要走本流程。

适用栈：{{STACK_LINE}}
策略：{{STRATEGY}}

## 总原则

- 先用确定性命令证明状态，再让 AI 辅助分析。
- 首轮只读：除非用户明确授权"修复/改代码/补测试"，只运行命令、只写报告，不改源码。
- 分层收敛：从后端 → 数据库 → 前端 → API 契约 → 前后端联调 → E2E，逐层把失败范围切小。
- 命令只用 `命令清单.json` 锁定的；缺命令就先报缺口，不临时发明。

## 执行步骤

```text
- [ ] 步骤 0：加载控制层（按 必读顺序.md）
- [ ] 步骤 1：指纹校验（_scripts/validate_testing_docs.py）
- [ ] 步骤 2：分层执行只读命令
- [ ] 步骤 3：服务在跑时做冒烟（按需）
- [ ] 步骤 4：按授权决定是否进修复循环
- [ ] 步骤 5：写报告到 07-测试报告
- [ ] 步骤 6：汇报结论
```

### 步骤 0：加载控制层

读 `00-控制层/测试画像.json`、`代码事实指纹.json`、`命令清单.json`、`报告结构.json`，明确本项目的栈、模块、分层和锁定命令。

### 步骤 1：指纹校验

运行：

```bash
python3 doc/testing/00-控制层/_scripts/validate_testing_docs.py <项目根> --check-drift
```

- 结构缺失：停下，提示重跑 `testing-init`。
- 指纹漂移（manifest/命令/版本变化）：告知用户"代码事实已变，建议重跑 testing-init"，由用户决定继续或刷新。
- 一致：继续。

### 步骤 2：分层执行只读命令

按 `命令清单.json` 的 `layers` 顺序执行每层 `readonly` 命令：

1. backend：编译 + 单测 + MockMvc/Repository 测试。
2. database：仅 test/local profile、Testcontainers 或临时 schema；不跑 DROP/TRUNCATE/全表 DELETE。
3. frontend：单测/组件测试、lint、typecheck、build。
4. api-contract / e2e：见步骤 3。

每层失败先记录第一个错误摘要，不立即修。

### 步骤 3：服务在跑时做冒烟（按需）

若后端/前端已在本地运行（用 `lsof` 确认端口）：执行 `命令清单.json` 各层的 `smoke` 命令。先做健康检查与无 console/page error，再测一个有意义的流程（登录、列表、表单、API 健康）。失败保留截图/trace/日志路径。

### 步骤 4：按授权决定是否进修复循环

仅当用户已明确授权修复：对目标命令最多循环 10 轮（运行 → 读第一个失败 → 最小修复 → 重跑），一次修一个错误；错误持续增多就停下换思路。不删测试变绿，不用忽略注释掩盖失败。

### 步骤 5：写报告

按 `报告结构.json` 写一份报告到 `07-测试报告/{date}/{time}-{mode}-测试报告.md`，不覆盖旧报告。只摘关键错误，长日志写路径不贴全文。

### 步骤 6：汇报结论

先说通过/失败数量与实际运行命令；失败项给文件/行号、摘要、最小修复建议；说明哪些层没跑及原因；最后给报告路径。
