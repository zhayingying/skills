# 测试报告归档

每轮测试结束后，在被测项目根目录写一份 Markdown 报告到 `doc/test/`。

## 路径规则

使用本地时间按日期建目录、按时分秒命名：

```text
doc/test/YYYY-MM-DD/HHmmss-<mode>-测试报告.md
```

示例：

```text
doc/test/2026-06-24/143205-vue3-unit-component-测试报告.md
doc/test/2026-06-24/151020-springboot-mysql-test-测试报告.md
doc/test/2026-06-24/160015-full-stack-vue3-springboot-mysql-测试报告.md
```

`<mode>` 使用 `recommendedStrategy.mode`。如果没有 mode，使用 `native-scripts`。

## 写入规则

- 可以创建 `doc/test/` 和当天日期目录。
- 每轮测试默认只新增一份时间戳报告，不覆盖旧报告。
- 不默认维护 `latest.md`；除非用户明确要求。
- 不自动删除、移动、压缩旧报告。
- 如果历史报告过多，只统计数量并建议清理，等用户确认后再处理。

## 报告内容

报告保持短而可执行：

```markdown
# 测试报告

- 时间：
- 项目：
- 策略：
- 范围：
- 结论：

## 执行命令

| 命令 | 结果 | 摘要 |
|---|---|---|

## 失败项

- 文件/接口：
- 错误摘要：
- 疑似原因：
- 建议修复：

## 未运行项

- 项目：
- 原因：

## 证据

- 截图/trace/日志路径：
```

不要把完整长日志粘进报告；只摘关键错误，并写明完整日志、截图、trace 的路径。

## 首轮只读边界

创建测试报告是允许的记录行为。除此之外，首轮仍然不要修改源码、测试、配置、lockfile、快照、依赖或生成物。
